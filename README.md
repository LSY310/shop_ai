## 🗓️ Day 1 - LLM 기반 쇼핑몰 챗봇 구현

### 🔍 문제 발생
- OpenAI API 대신 Google Gemini 모델(Free Tier) 사용 결정.

- 초기 연동 시 모델 이름 설정 오류로 인해 API 호출 실패.

### 🛠️ 해결 과정
- 공식 문서 확인 후, 실제 사용 가능한 모델 목록을 `curl` 명령어로 직접 조회하여 정확한 모델 경로 확인.

```bash
# 사용 가능한 모델 목록 직접 확인
curl "[https://generativelanguage.googleapis.com/v1/models?key=YOUR_API_KEY](https://generativelanguage.googleapis.com/v1/models?key=YOUR_API_KEY)"

#응답에서 확인한 정확한 모델명("models/gemini-2.5-flash")을 코드에 적용하여 해결
```

### 💡 배운 점
- 엄격한 모델 검증: Google Gemini API는 모델 ID가 정확해야만 작동함.

- 직접 검증의 중요성: 때로는 문서보다 실제 API 응답을 확인하는 것이 가장 빠르고 확실함.

- LLM 구조 이해: API Key → Model Selection → Generate Content로 이어지는 호출 흐름 파악 완료.

## 🗓️ Day 2 - RAG 시스템 구축 및 검색 알고리즘 고도화

### 🔍 목표
- 쇼핑몰 운영 정책(data.txt)을 기반으로 한 지능형 QA 시스템 구현
- 단순 키워드 검색의 한계를 극복하고 의미 기반(Semantic) 검색으로 성능 개선.

### 🛠️ 해결 과정
- 1단계: Keyword Matching 기반 RAG (초기 모델)

    방식: 사용자의 질문을 단어 단위로 나누고, 해당 단어가 포함된 문장을 문서에서 추출하여 Gemini에게 전달.

    문제점: 사용자가 "택배비 얼마야?"라고 물었을 때, 문서에 "배송비"라는 단어만 있으면 관련 정보를 찾지 못하는 **'동의어 미매칭'** 문제 발생.

- 2단계: Embedding & Vector DB 기반 RAG (고도화 모델)

    해결: Gemini Embedding Model ("gemini-embedding-001")을 도입하여 텍스트를 고차원 벡터로 변환.

- Vector DB: ChromaDB를 사용하여 정책 데이터를 벡터화하여 저장 및 검색.

    결과: 단어가 일치하지 않아도 "택배비"와 "배송비", "돈 돌려줘"와 "환불"의 의미적 유사도를 계산해 관련 정보를 정확히 추출함.

### 💡 배운 점
- RAG의 핵심: 단순히 문서를 읽어주는 것이 아니라, 질문의 '의도'와 가장 가까운 '데이터 조각'을 찾아주는 검색 엔진의 성능이 답변의 퀄리티를 결정함.

- Vector DB 활용: 비정형 데이터를 효율적으로 관리하고 검색하기 위한 벡터 데이터베이스의 구조와 활용법 습득.

## 🗓️ Day 3 - MVP 엔진 완성(Function Calling을 이용한 지능형 에이전트 구현)

### 🔍 목표
- 단순 답변을 넘어 쇼핑몰의 실시간 데이터(주문, 상품)를 직접 다루는 에이전트(Agent) 시스템 구축.
- 정형 데이터(SQL)와 비정형 데이터(Vector)를 동시에 활용하는 하이브리드 지식 체계 완성.

### 🛠️ 해결 과정
- 하이브리드 데이터베이스 아키텍처 설계

    정형 데이터(SQLite): 주문 번호, 배송 상태처럼 정확한 수치와 상태 값이 필요한 정보는 관계형 데이터베이스로 관리.

    비정형 데이터(ChromaDB): 상품 설명, 추천 사유 등 의미 기반 검색이 필요한 정보는 벡터 데이터베이스로 관리.

- Function Calling(도구 사용) 도입

    Gemini에게 get_order_status와 search_and_recommend라는 Python 함수(도구)를 제공.

    작동 원리: LLM이 질문을 분석하여 스스로 최적의 도구를 선택하고, 파라미터(주문번호 등)를 추출하여 함수를 실행한 뒤 그 결과를 바탕으로 최종 답변을 구성함.

- 검색 노이즈 필터링 및 페르소나 강화

    문제점: 데이터 부족 시 벡터 검색이 관련 없는 상품(예: 옷 요청에 마우스 추천)을 반환하는 현상 발생.

    해결: System Instruction을 고도화하여 LLM이 도구로부터 전달받은 로우 데이터 중 질문의 의도와 일치하는 항목만 선별하여 답변하도록 2차 추론 필터링 적용.

### 🚨 Troubleshooting: API Quota Exceeded (HTTP 429 Error)
현상: 반복적인 기능 테스트 및 도구 호출로 인해 Gemini API 할당량 초과 발생.

대응: 지수 백오프(Exponential Backoff) 개념을 학습하고, 테스트 간격 조정 및 도구 단위 유닛 테스트를 통해 API 호출 효율화.

### 💡 배운 점
- Agentic Workflow: LLM은 더 이상 수동적인 답변기가 아니라, 외부 도구를 적재적소에 활용해 문제를 해결하는 '실행 주체'가 될 수 있음을 체득함.

- 데이터 적재적소 배치: 모든 데이터를 RAG로 처리하기보다, 상태값(주문)은 SQL로, 취향(추천)은 Vector DB로 나누어 관리하는 것이 시스템 안정성과 정확도 면에서 훨씬 유리함을 학습.

- Prompt Engineering의 중요성: 도구의 이름과 설명(Docstring)을 얼마나 구체적으로 작성하느냐가 LLM의 도구 선택 정확도를 결정짓는 핵심 변수임을 깨달음.

## 🗓️ Day 4 - 매출 분석 및 경영 제언 기능 추가 (Pandas 기반 자동화)

### 🔍 목표
- 단순 주문 조회를 넘어, 전체 데이터를 통계적으로 분석하여 경영 인사이트를 제공하는 BI(Business Intelligence) 기능 구현.

- Pandas 라이브러리를 도입하여 복잡한 데이터 집계 로직의 확장성과 처리 속도 개선.

### 🛠️ 해결 과정
- 1단계: 데이터 스키마 확장 및 샘플링

    매출액(price)과 주문일자(order_date) 컬럼을 추가하여 분석 가능한 정형 데이터 구조로 개선.

    실질적인 분석 흐름을 확인하기 위해 15건의 가상 주문 데이터를 구축하여 데이터 밀도 확보.

- 2단계: Pandas 기반 매출 분석 자동화 (analyze_sales_report)

    sqlite3로 추출한 데이터를 Pandas DataFrame으로 변환하여 총 매출, 객단가(평균 주문 금액), 베스트셀러 상품을 1차 가공.

    SQL 쿼리만으로는 복잡한 통계 계산을 파이썬 코드 한 줄로 처리하여 백엔드 로직의 가독성 증대.

- 3단계: LLM 기반 데이터 해석 및 전략 제언

    가공된 통계 수치를 Gemini에게 전달하고, 시스템 지침(System Instruction)을 통해 단순 숫자 나열이 아닌 '데이터 기반의 경영 조언'을 생성하도록 고도화.

    결과 예시: "현재 냉감 소재 의류 판매가 급증하고 있으니 관련 재고를 20% 이상 선확보하십시오."

### 💡 배운 점
- 데이터 기반 의사결정(Data-Driven): AI가 단순히 텍스트를 생성하는 존재가 아니라, 수치 데이터를 바탕으로 비즈니스 전략을 제안할 수 있는 '분석가' 역할을 수행할 수 있음을 체득.

- Pandas 활용의 이유: 데이터 규모가 커질수록 SQL 직접 조회보다 Pandas를 통한 메모리 내 연산이 복잡한 통계 분석과 LLM 연동에 훨씬 유연하다는 것을 이해함.

- 에이전트 확장성: 기존 기능(주문/추천)을 해치지 않고 새로운 도구(분석)를 추가하는 것만으로도 서비스의 성격(B2C → B2B)을 확장할 수 있는 에이전트 설계 구조의 강점을 경험.

## 🗓️ Day 5 - 콘텐츠 자동 생성 및 엑셀 일괄 등록 구현

### 🔍 목표
- 네이버 스마트스토어의 등록 절차를 간소화하기 위한 자동화 로직 설계.

- 사용자 키워드 기반의 SEO(검색 최적화) 상품명 및 상세 설명 생성 프롬프트 구축.

- 생성된 데이터를 실제 네이버 양식에 맞춰 엑셀 파일로 자동 변환하는 파이프라인 형성.

### 🛠️ 해결 과정
- 1단계: 네이버 스마트스토어 등록 프로세스 및 양식 분석

    개별 등록의 번거로움을 해결하기 위해 '상품 일괄등록' 기능을 분석하고 필수 엑셀 컬럼 파악.

    모바일 노출에 유리한 '사진+글' 구조의 HTML 상세페이지 양식 설계.

- 2단계: RAG 및 Few-shot 기반 콘텐츠 생성 고도화

    기존 상품 DB를 참고하여 브랜드 고유의 톤앤매너를 유지하면서 신규 상품 설명 생성.

    SEO 최적화 알고리즘을 프롬프트에 녹여 검색 품질 체크를 통과할 수 있는 상품명과 태그 추출.

- 3단계: Pandas 기반 엑셀 자동 생성 엔진 구현

    AI가 생성한 텍스트 데이터를 Pandas를 활용하여 네이버 전용 엑셀 양식(.xlsx)으로 즉시 변환.

    운영자가 수동으로 데이터를 옮겨 적는 과정 없이 파일 업로드만으로 등록이 가능하도록 프로세스 최적화.

### 💡 배운 점
- 데이터 호환성 확보: 백엔드에서 생성된 비정형 데이터(텍스트)를 정형 데이터(엑셀)로 변환하여 외부 플랫폼과 연동하는 데이터 엔지니어링의 기초를 체득함.

- 실무적 문제 해결: '귀찮음'이라는 실질적인 운영상의 문제를 분석하여 기술적 대안(엑셀 자동 생성)으로 해결하며 개발자의 비즈니스 기여도를 체감함.

- 프롬프트 엔지니어링: HTML 태그를 포함한 구조화된 출력을 유도하여 AI 결과물을 서비스에 즉시 가공 없이 투입할 수 있는 효율성을 확보함.

## 🗓️ Day 6 - AI 라우터(Router) 구축 및 에이전트 역할 분리 최적화

### 🔍 목표
- 흩어져 있는 RAG(정책 검색), DB 조회, 데이터 분석, 콘텐츠 생성 기능을 하나로 통합하는 중앙 관제 시스템(Orchestrator) 구축.

- 질문의 의도를 사전에 분류하여 불필요한 도구 호출을 줄이고, 각 작업에 최적화된 실행 경로(Routing) 설계.


### 🛠️ 해결 과정
- 1단계: 관심사 분리 설계

    라우터(Router): 질문이 '정책(POLICY)', '기능 실행(ACTION)', '일반 대화(GENERAL)' 중 어디에 속하는지 판단하여 문을 열어주는 역할만 전담.

    에이전트(Agent): 라우터가 열어준 문 안에서 system_instruction에 따라 실제 도구(Function Calling)를 사용하고 대화의 맥락을 관리하는 실행 역할 전담.

- 2단계: 지능형 워크플로우 구현

    system_instruction 고도화를 통해 도구 간의 연쇄 동작 설계.

    예: 상품 등록 요청 시 generate_smartstore_content로 문구를 먼저 생성하고, 사용자의 긍정적인 피드백 확인 시 즉시 save_to_db를 호출하여 DB에 자동 저장하는 흐름 구축.

- 3단계: 데이터 처리 최적화 및 안정성 확보

    analyze_sales_report 시 전체 데이터를 모델에 던지는 대신, Pandas로 미리 계산된 요약 수치와 최근 5건의 샘플 데이터만 전달하도록 로직을 수정하여 토큰 효율성 및 분석 정확도 향상.

    라우터의 JSON 파싱 오류나 오분류 발생 시 자동으로 기본 에이전트 세션으로 연결되는 Fallback(방어 코드) 로직 적용.

### 💡 배운 점
- 설계의 중요성: 개별 기능을 구현하는 것보다, 각 기능을 어떤 순서로 배치하고 역할을 어떻게 나누느냐가 전체 시스템의 지능(Intelligence)을 결정한다는 것을 깨달음.

- 프롬프트 엔지니어링의 정교화: Docstring은 도구의 설명서로, System Instruction은 에이전트의 업무 지침으로 활용하여 LLM이 스스로 판단하고 행동하게 만드는 '자율적 에이전트' 구조를 체득함.

- 효율적인 자원 관리: 라우팅을 통해 문서 검색(RAG)이 필요한 시점과 데이터 분석(Pandas)이 필요한 시점을 명확히 구분함으로써 처리 속도와 정확도를 동시에 잡는 아키텍처를 경험함.

## 🗓️ Day 7 - End-to-End 자동화 엔진 완성 (콘텐츠 생성부터 API 등록까지)

### 🔍 목표
- LLM이 생성한 상품 콘텐츠를 단순 확인하는 것을 넘어, 실제 시스템(DB/API)에 한 번에 등록하는 완결형 에이전트 구축.

- 네이버 스마트스토어 API 권한 제한을 고려하여, '엑셀용 DB 저장'과 '가상 내부 시스템 API 전송'이라는 투트랙 자동화 로직 설계.


### 🛠️ 해결 과정
- 1단계: Tool 추가 및 확장

    register_to_internal_system 함수를 추가하여, 생성된 JSON 데이터를 외부 FastAPI 서버로 전송하는 파이프라인 구축.

    기존 save_to_db는 엑셀 대량 등록을 위한 백업 용도로 유지하여 실무 적합성 확보.

- 2단계: 에이전틱 워크플로우(Agentic Workflow) 설계

    system_instruction을 고도화하여 LLM이 [콘텐츠 생성] → [사용자 컨펌] → [DB 저장 및 API 전송]의 순서를 스스로 판단하고 실행하도록 설계.

    chat_session의 대화 맥락 유지 기능을 활용해 사용자가 "등록해줘"라고만 해도 이전 단계의 상품 데이터를 기억해 전송하도록 구현.

- 3단계: 가상 API 서버(Mock Server)를 통한 검증

    FastAPI를 이용해 상품 데이터를 수신하는 가상 엔드포인트(post /api/v1/internal/register)를 구축하여 데이터 전송 및 수신 여부 확인.

### 💡 배운 점
- 에이전트의 완결성: AI가 단순히 텍스트를 보여주는 '답변기'를 넘어, 외부 API와 통신하여 실무적인 액션을 수행하는 '실행 주체'가 될 때 서비스의 가치가 극대화됨을 체득함.

- Mock 시스템의 중요성: 외부 권한 이슈(네이버 API 등)로 개발이 막힐 때, 가상 API 서버를 구축하여 전체 로직을 검증하는 방식이 백엔드 개발자에게 얼마나 중요한 역량인지 깨달음.

- 효율적인 자원 관리: 라우팅을 통해 문서 검색(RAG)이 필요한 시점과 데이터 분석(Pandas)이 필요한 시점을 명확히 구분함으로써 처리 속도와 정확도를 동시에 잡는 아키텍처를 경험함.

## 🗓️ Day 8 - 인터페이스 고도화 및 사용자 경험(UX) 최적화

### 🔍 목표
- 기존 시스템은 도구(Tool) 실행 결과인 Raw JSON 데이터가 그대로 출력되어 가독성이 떨어졌음.

- API 호출 및 로직 처리 중 발생하는 대기 시간 동안 사용자에게 피드백이 없어 시스템이 멈춘 듯한 인상을 줄 위험이 있었음.

- 백엔드 개발자로서 데이터의 정확성뿐만 아니라, 사용자에게 도달하는 최종 접점의 경험(UX)까지 고려한 설계가 필요하다고 판단

### 🛠️ 해결 과정
- 1단계: 페르소나 기반 응답 정제

    시스템 프롬프트를 통해 '친절한 쇼핑몰 비서' 역할을 부여하고, LLM이 모든 기술적 응답을 자연스러운 문장으로 요약하여 보고하도록 로직 수정

- 2단계: Streamlit 기반 채팅 UI 구축

    st.chat_message와 st.chat_input을 도입하여 현대적인 메시징 인터페이스로 전면 개편.

- 3단계: 상태 표시기(Status Indicator) 도입

    st.status를 활용하여 ‘의도 분석 → 시스템 조회 → 요약 완료’의 백엔드 처리 단계를 사용자에게 실시간 시각화.

- 4단계: 예외 처리 Fallback
    API 쿼터 초과(429) 에러 발생 시 프로그램이 종료되지 않고, 사용자에게 상황을 친절히 안내하며 재시도를 유도하는 예외 처리 로직 구현.

### 💡 배운 점
- UX 중심 사고: 기술적으로 완벽한 로직만큼이나, 처리 과정을 사용자에게 친절히 공유하는 인터페이스가 시스템의 신뢰도를 결정한다는 것을 체감함.

- 성능 및 안정성: 적절한 에러 핸들링을 통해 시스템 중단 시간을 최소화하고, 할당량 부족 상황에서도 서비스 연속성을 유지하는 안정적인 구조를 설계함.

- 성과: 직관적인 UI/UX 개선을 통해 초기 버전 대비 조작 편의성을 대폭 향상시켰으며, 면접 시 "사용자 경험을 고민하는 백엔드 개발자"로서의 강점을 어필할 수 있는 기반을 마련함.
